# Руководство по разработке JoeCamelBot

Мы придерживаемся структурированного процесса разработки для обеспечения качества кода. Пожалуйста, следуйте данным правилам при внесении изменений.

## Процесс разработки

1. Создание задачи (Issue): Любые изменения должны начинаться с создания тикета с описанием бага или новой функции.
2. Ветки разработки: Создавайте отдельную ветку для каждой задачи. Прямые коммиты в ветку main запрещены.
   - Формат именования: feature/название или fix/название.
3. Рабочие деревья (Worktrees): Рекомендуется использовать git worktree для параллельной работы над несколькими задачами.
4. Pull Requests: Изменения вносятся через Pull Request (PR) из вашей ветки в main с использованием GitHub CLI (gh).

## Стандарты кодирования

- Асинхронность: Используйте async и await для всех операций ввода-вывода (запросы к API, работа с БД).
- Типизация: Указывайте типы аргументов и возвращаемых значений для всех функций.
- Документирование: Пишите docstrings в стиле Google для всех нетривиальных функций и классов.
- Обработка ошибок: Используйте блоки try/except для сетевых и дисковых операций, логируйте ошибки через модуль logging. Избегайте использования print().

## Рецензирование кода (Code Review)

- Каждый PR должен пройти проверку перед слиянием.
- Агенты ИИ обязаны запрашивать проверку человеком перед тем, как пометить PR готовым к слиянию.
- Перед слиянием убедитесь, что все автоматические проверки выполнены успешно.

## Управление Docker

- Не выполняйте команды docker-compose up или down в автоматическом режиме агента; жизненный цикл контейнеров управляется пользователем.
- Если изменения требуют пересборки образа, укажите это в описании к Pull Request.
